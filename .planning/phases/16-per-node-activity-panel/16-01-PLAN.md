<objective>
Complete the per-node activity panel (FLOW-13) by extending the existing `NodeDetailPanel` so that clicking any canvas node shows up to 10 relevant activity events — including nodes that currently return empty (qdrant, obsidian, knowledge-curator, per-agent nodes). Add AbortController cleanup to the heartbeat fetch, trim event cap from 15 to 10, and render a documented "Limited activity data" note for nodes with inherently sparse coverage.

Purpose:
- Close the last FLOW track gap in v1.3 before the riskier Phase 17 canvas refactor.
- Eliminate broken-feeling empty panels on valid nodes (qdrant, obsidian, agents) by matching events via keyword mapping rather than strict `event.node === nodeId` equality only.
- Fix a real resource leak: current `useEffect` fires `fetch('/api/heartbeat')` on every `nodeId` change but never aborts in-flight requests, causing `setState` on stale promises and wasted network when users click nodes rapidly.
- Be honest about sparse data: nodes with no instrumentation get a documented indicator, not a fake feed.

Output:
- New pure module `src/lib/node-keyword-map.ts` with `matchEventsForNode(nodeId, events)` and `isSparseNode(nodeId)`, fully unit-tested.
- Updated `src/components/flow/node-detail-panel.tsx` using the module, an AbortController, a 10-event cap, and a sparse indicator.
- Vitest tests: one for the pure mapping (no React), one for the panel behavior (AbortController + sparse note + 10-cap).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → `/api/heartbeat` | `agentId` query param is user-controlled (click source is a canvas node, but any client-side JS can request arbitrary values). Existing allowlist `^[a-zA-Z0-9_-]+$` in the route handles this. |
| browser → `/api/activity` | Read-only, no query params currently. |
| node-keyword-map regexes | Regexes run against `event.message` strings sourced from the activity API (which reads `APO_CRON_LOG`, `HEARTBEAT.md`, `skill-contributions.jsonl`, `PROPOSALS_PATH`). If an attacker controls those files, they control `message`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-01 | Tampering | node-keyword-map regexes | mitigate | All regexes use bounded `\b` anchors or specific literals. No greedy `.*` that could ReDoS on adversarial input. Review the `tunnels`/`obsidian` regexes during PR — reject any regex containing nested quantifiers. |
| T-16-02 | Denial of Service | AbortController leak | mitigate | The whole point of Task 2 step 3 — cleanup aborts every in-flight fetch. Without this, rapid node-click spams heartbeat requests. Directly addressed by FLOW-13 success criterion 3. |
| T-16-03 | Information Disclosure | AbortError swallowed too broadly | accept | We only swallow when `err.name === 'AbortError'`. Other errors still set `heartbeatContent(null)` — same behavior as today. Low risk: we are not logging raw errors anywhere new. |
| T-16-04 | Spoofing | `nodeId` passed to panel | accept | `nodeId` comes from `onNodeClick` wired to React Flow — not from URL or user input. No new attack surface. |
| T-16-05 | Repudiation | N/A | accept | Read-only client UI change — no state mutation, nothing to repudiate. |
| T-16-06 | Elevation of Privilege | N/A | accept | Client-only code change. No API or permission changes. |
