Step 2 — Implement src/lib/content-scanner.ts:

Export interfaces ScanMatch (patternName: string; severity: 'HIGH' | 'MEDIUM'; redacted: string)
and ScanResult (blocked: boolean; matches: ScanMatch[]; cleanContent: string).

Export PATTERNS array of 18 entries, each { name: string; pattern: RegExp; severity: 'HIGH' | 'MEDIUM' }.
All RegExp literals must NOT use the 'g' (global) flag — use non-global flags only.

The 18 patterns:
1. aws_access_key — HIGH — matches AKIA followed by 16 uppercase alphanumeric chars
2. aws_secret_key — HIGH — matches aws near secret/key near a 40-char base64 string in quotes, case-insensitive
3. github_token_pat — HIGH — matches 'ghp_' prefix plus 36 alphanumeric chars
4. github_token_oauth — HIGH — matches 'gho_' prefix plus 36 alphanumeric chars
5. github_token_server — HIGH — matches 'ghs_' prefix plus 36 alphanumeric chars
6. pem_private_key — HIGH — matches PEM private key header line (BEGIN ... PRIVATE KEY)
7. jwt_token — HIGH — matches three base64url segments separated by dots (eyJ prefix pattern)
8. credit_card — HIGH — matches Visa/MC/Amex/Discover card number formats using standard BIN ranges
9. ssn_us — HIGH — matches DDD-DD-DDDD hyphenated US Social Security Number format
10. password_in_url — HIGH — matches credential-bearing URLs with user:pass@host pattern
11. slack_webhook — HIGH — matches full Slack incoming webhook URL pattern (hooks.slack.com/services/T.../B...)
12. xss_script_tag — HIGH — matches opening script tag (case-insensitive) as XSS injection vector
13. shell_injection — HIGH — matches semicolons before common shell commands or backtick command substitution
14. email_address — MEDIUM — standard email format (letters/digits/dots/plus @domain)
15. phone_us — MEDIUM — US phone number in common formats including optional international prefix
16. generic_secret_assign — MEDIUM — key=value pairs where key contains password/secret/api_key/token and value is 8+ quoted chars, case-insensitive
17. generic_long_token — MEDIUM — generic 40+ alphanumeric character blobs (many false positives expected; flag only)
18. sql_injection_union — MEDIUM — classic SQL injection keywords: UNION SELECT, DROP TABLE, INSERT INTO, DELETE FROM (case-insensitive)

scanContent(text: string): ScanResult implementation:
1. Null-safe coerce: text = text ?? ''
2. Length guard: if text.length > 4096 return { blocked: false, matches: [], cleanContent: text }
3. For each pattern in PATTERNS: wrap in try/catch; call pattern.exec(text); if match found,
   push { patternName: name, severity, redacted: found[0].slice(0, 8) + '...' } to matches array
4. Build cleanContent: start with text; for each HIGH match found, replace using text.replace(pattern, '[REDACTED]')
   (non-global replace replaces first occurrence; one call per pattern is the intended design)
5. Return { blocked: matches.some(m => m.severity === 'HIGH'), matches, cleanContent }

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| agent→POST /api/hive | Agent-generated content enters here; untrusted text that may contain secrets or injection payloads |
| agent→POST /api/recall/ingest | Ingestion trigger; low-risk (no agent content body, fires ingestAllSessions) |
| agent→POST /api/memory-consolidate | Consolidation trigger; low-risk (no agent content body) |
| client→GET /api/audit-log | Reads audit data; single-user local tool, no auth required |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-24-01 | Information Disclosure | POST /api/hive — agent summary field | mitigate | scanContent() intercepts all outbound text; HIGH-severity patterns redact and block; MEDIUM patterns flag; matched values stored as truncated preview only (first 8 chars + '...'), never full value stored in DB |
| T-24-02 | Tampering | POST /api/hive — XSS via script tag in summary | mitigate | xss_script_tag pattern at HIGH severity — blocked before DB write, event logged to audit_log |
| T-24-03 | Tampering | writeAuditLog() — SQL injection via actor/target fields | mitigate | writeAuditLog() uses better-sqlite3 parameterized .run() with named parameters — same protection as all existing routes (VERIFIED: T-20-03 in 20-01-SUMMARY.md) |
| T-24-04 | Denial of Service | scanContent() — catastrophic backtracking on crafted long input | mitigate | Length cap of 4096 chars before any pattern evaluation; try/catch per-pattern execution; all patterns use non-backtracking design (no nested quantifiers) |
| T-24-05 | Repudiation | Audit write failure silently hiding a significant action | accept | try/catch logs to console.error; primary action succeeds regardless; acceptable for single-user local tool threat model |
| T-24-06 | Elevation of Privilege | shell_injection pattern — shell command injection via agent summary | mitigate | shell_injection pattern at HIGH severity blocks and redacts content containing backtick substitution or semicolon-prefixed shell commands |
</threat_model>
